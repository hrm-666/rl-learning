# 多臂老虎机学习笔记

这篇笔记同样是用于自己学习所用。其实前一周已经正式开始了强化学习的学习，但是有点眼高手低了，直接往后面正式算法上尝试了，但仔细回想，现在也并不着急还是先把基础打好后面再说。所以今天的主题便是强化学习领域最最基础入门的多臂老虎机问题。

后面大概是以这个主线来走：
- 多臂老虎机问题到底在描述什么
- 什么是探索和利用的矛盾
- 累计懊悔值为什么是一个重要指标
- `epsilon-greedy`、衰减 `epsilon-greedy`、`UCB`、`Thompson Sampling` 各自在做什么
- 这些方法和后面强化学习主线之间是什么关系

## 1. 多臂老虎机在解决什么问题

多臂老虎机可以理解成这样一个场景：

- 面前有 `K` 根拉杆
- 每根拉杆被拉一下，都有一定概率得到奖励
- 但是我一开始并不知道每根拉杆的真实中奖概率

目标不是“每一步都一定拿最大奖励”，而是：

`在不断尝试的过程中，尽量多拿奖励，并尽快找到最优拉杆`

如果把它抽象一点看，多臂老虎机研究的是：

`在信息不完整的情况下，怎样一边试、一边学、一边做决策`

这也是强化学习里非常经典的入门模型。

## 2. 为什么它适合做强化学习入门

多臂老虎机虽然简单，但它把强化学习里最核心的矛盾单独拿了出来：

- 利用：选当前看起来最好的动作
- 探索：去尝试那些还不确定、但可能更好的动作

这个问题为什么简单？正是因为它大大简化了正常强化学习问题里面的“状态”，可以说完全就没考虑这一重要因素。

- 没有状态转移
- 没有长期路径规划
- 没有复杂环境动态

所以它非常适合先把“探索和利用”的逻辑理解清楚。

可以把它看成是：

`先学会怎么在不确定中做选择，再去学有状态转移的强化学习问题`

## 3. Bernoulli Bandit 是什么

在我的代码里，环境写成了 `BernoulliBandit`：

```python
class BernoulliBandit:
    def __init__(self, k):
        self.probs = np.random.uniform(size=k)
```

这里的意思是：

- 一共有 `k` 根拉杆
- 每根拉杆都有一个 `0~1` 之间的中奖概率
- 这个概率固定，但智能体事先不知道

当执行：

```python
bandit.step(k)
```

时，本质上是在做一次伯努利试验：

- 以 `self.probs[k]` 的概率返回 `1`
- 否则返回 `0`

所以每根拉杆都可以理解成一个“奖励服从伯努利分布”的动作。

## 4. 多臂老虎机里没有状态，只有动作

这是和后面 Q-learning、DQN 很不一样的一点。

在多臂老虎机里：

- 智能体每一步面对的环境都一样
- 没有 `state -> next_state` 的变化
- 决策只是在多个动作里选一个

所以这里学到的不是：

`某个状态下哪个动作更好`

而是更简单的：

`哪根拉杆整体上更值得长期选择`

因此，多臂老虎机可以理解成“只有一个隐含状态的强化学习问题”。

## 5. 这类问题的核心矛盾：探索和利用

如果我永远只选当前估计最好的拉杆，那就是利用。

好处是：

- 短期内看起来更稳

坏处是：

- 一开始估计可能是错的
- 可能会过早卡在次优拉杆上

如果我经常随机尝试别的拉杆，那就是探索。

好处是：

- 有机会发现更优拉杆

坏处是：

- 会浪费一些机会在明显不好的拉杆上

所以多臂老虎机本质上一直在回答一个问题：

`我到底该把多少机会拿去试错，又该把多少机会拿去赚钱`

## 6. 奖励和期望奖励要分开理解

这个问题里很容易把“单次奖励”和“拉杆好坏”混在一起。

要分清：

- 单次奖励 `r`：这一次拉动后拿到的是 `0` 还是 `1`
- 真实概率 `p`：这根拉杆长期平均能带来多大回报
- 估计值 `estimate`：我目前根据历史经验，对这根拉杆中奖概率的估计

例如：

- 一根中奖概率 `0.8` 的拉杆，这次也可能没中奖
- 一根中奖概率 `0.2` 的拉杆，这次也可能正好中奖了

所以算法不能因为一两次结果就下结论，而要不断更新长期估计。

## 7. 奖励更新机制：从数学公式到代码

这一部分是多臂老虎机里最值得真正搞懂的地方之一。

在代码里，我们会不断维护每根拉杆奖励概率的估计值。这个估计值不是拍脑袋更新，而是严格从样本均值一步一步推出来的。

### 7.1 先写出完整的数学推导

假设某一根拉杆已经被拉了 `k` 次，对应观测到的奖励是：

`r_1, r_2, ..., r_k`

那么第 `k` 次更新后的估计值定义为前 `k` 次奖励的平均值：

```math
Q_k = \frac{1}{k}\sum_{i=1}^{k} r_i
```

把最后一次奖励单独拆出来：

```math
Q_k = \frac{1}{k}\left(r_k + \sum_{i=1}^{k-1} r_i\right)
```

而前 `k-1` 次的平均奖励是：

```math
Q_{k-1} = \frac{1}{k-1}\sum_{i=1}^{k-1} r_i
```

所以：

```math
\sum_{i=1}^{k-1} r_i = (k-1)Q_{k-1}
```

代回上式：

```math
Q_k = \frac{1}{k}\left(r_k + (k-1)Q_{k-1}\right)
```

接着把 `(k-1)Q_{k-1}` 改写成 `kQ_{k-1} - Q_{k-1}`：

```math
Q_k = \frac{1}{k}\left(r_k + kQ_{k-1} - Q_{k-1}\right)
```

最后整理成递推形式：

```math
Q_k = Q_{k-1} + \frac{1}{k}\left(r_k - Q_{k-1}\right)
```

这就是你图里那条最关键的推导链。

### 7.2 这个递推公式到底在说什么

这个式子可以读成：

`新估计 = 旧估计 + 学习步长 × (新样本 - 旧估计)`

也就是：

```math
Q_k = Q_{k-1} + \frac{1}{k}(r_k - Q_{k-1})
```

其中：

- `Q_{k-1}` 是这根拉杆之前的平均奖励估计
- `r_k` 是这次新观察到的奖励
- `r_k - Q_{k-1}` 是这次样本相对于旧估计的误差
- `1/k` 决定这次误差要拿多少比例来修正旧估计

所以它不是随便凑出来的经验公式，而是样本均值公式的等价递推写法。

### 7.3 为什么要把均值公式改写成递推公式

如果直接按照定义算：

```math
Q_k = \frac{1}{k}\sum_{i=1}^{k} r_i
```

那么每次更新都要：

- 把历史所有奖励重新保存下来
- 再重新求一次和

这样做当然可以，但不适合写成在线学习代码。

而递推形式：

```math
Q_k = Q_{k-1} + \frac{1}{k}(r_k - Q_{k-1})
```

只需要知道三样东西：

- 旧估计 `Q_{k-1}`
- 这次奖励 `r_k`
- 当前是第几次更新 `k`

所以它特别适合边交互边更新的强化学习实现。

### 7.4 数学符号如何对应到代码

在 `mab.py` 里，更新奖励估计的代码是：

```python
self.estimates[k] += 1. / (self.counts[k] + 1) * (r - self.estimates[k])
```

这行代码和上面的数学递推式是一一对应的。

先把数学式再写一遍：

```math
Q_k = Q_{k-1} + \frac{1}{k}(r_k - Q_{k-1})
```

对应关系如下：

- `self.estimates[k]` 对应旧估计 `Q_{k-1}`
- `r` 对应当前新奖励 `r_k`
- `self.counts[k] + 1` 对应当前这次更新的样本数 `k`

于是代码其实就是在做：

```python
Q_new = Q_old + (1 / k) * (r - Q_old)
```

也就是把数学递推式原封不动地翻译成了程序。

### 7.5 为什么代码里要写 `self.counts[k] + 1`

这里的 `+1` 不是数学上多出来的，而是由代码执行顺序决定的。

在 `run()` 里，顺序是：

1. 先调用 `run_one_step()`
2. 在 `run_one_step()` 里拿到奖励 `r` 并更新估计值
3. 返回动作编号
4. 回到 `run()` 后再执行 `self.counts[k] += 1`

所以在执行这句更新代码的那一刻：

```python
self.estimates[k] += 1. / (self.counts[k] + 1) * (r - self.estimates[k])
```

`self.counts[k]` 记录的还是“这次之前已经试过多少次”。

如果这次实际上是第 `k` 次更新，那么程序此刻拿到的是：

- 旧次数 `k-1`

因此必须写成：

- `self.counts[k] + 1`

这样才和数学公式里的分母 `k` 对上。

### 7.6 用一个具体例子把公式和代码连起来

假设某根拉杆前两次奖励分别是：

`1, 0`

那么前两次的平均值是：

```math
Q_2 = \frac{1 + 0}{2} = 0.5
```

如果第 3 次奖励是 `1`，那么按递推公式：

```math
Q_3 = Q_2 + \frac{1}{3}(r_3 - Q_2)
```

代入数值：

```math
Q_3 = 0.5 + \frac{1}{3}(1 - 0.5)
    = 0.5 + \frac{1}{6}
    = \frac{2}{3}
```

而如果直接按平均值定义计算：

```math
Q_3 = \frac{1 + 0 + 1}{3} = \frac{2}{3}
```

两者完全一致。

所以代码里的更新写法并不是近似，而是把样本均值公式改写成了一个更适合程序实现的增量形式。

## 8. 什么是懊悔值

这份代码里一个很重要的指标是 `regret`：

```python
self.regret += self.bandit.best_prob - self.bandit.probs[k]
```

懊悔值不是“我这一步有没有中奖”，而是：

`我这一动作的期望收益，和最优动作的期望收益之间差了多少`

如果当前最优拉杆中奖概率是 `0.9`，我却选了一个 `0.6` 的拉杆，那么这一步的懊悔就是：

```python
0.9 - 0.6 = 0.3
```

累计起来，就是累计懊悔值。

## 9. 为什么累计懊悔值很重要

累计懊悔值衡量的是：

`因为没有一直选最优动作，我总共损失了多少期望收益`

它很适合评估 bandit 算法，因为：

- 它直接反映“试错成本”
- 它不受单次中奖随机波动影响太大
- 它更能说明算法有没有快速找到最优拉杆

如果一个算法累计懊悔增长很快，通常说明：

- 它探索得太乱
- 或者它长期卡在次优动作上

如果累计懊悔增长很慢，通常说明：

- 它比较快地找到了更优拉杆

## 10. `epsilon-greedy` 在做什么

`epsilon-greedy` 是最经典也最直观的策略：

- 以 `epsilon` 的概率随机探索
- 以 `1 - epsilon` 的概率选择当前估计值最大的拉杆

在你的代码里：

```python
if np.random.rand() < self.epsilon:
    k = np.random.randint(0, self.bandit.k)
else:
    k = np.argmax(self.estimates)
```

它的优点是：

- 简单
- 容易实现
- 很适合入门理解探索和利用

它的缺点是：

- 探索是“无差别随机探索”
- 即使已经很确定某些臂很差，仍然可能反复试到它们

## 11. 为什么固定 `epsilon` 可能表现很差

如果 `epsilon` 固定不变，会有两个典型问题。

第一个问题是：

- `epsilon` 太小，前期探索不够
- 容易过早相信某根次优臂
- 后面长时间卡住

第二个问题是：

- `epsilon` 太大，后期探索太多
- 明明已经找到好臂了，还总在随机乱试

所以固定 `epsilon` 经常会面临一个尴尬：

`前期和后期对探索的需求其实不一样，但固定 epsilon 没法同时兼顾`

## 12. 衰减 `epsilon-greedy` 为什么更自然

衰减 `epsilon-greedy` 的想法是：

- 刚开始信息少，多探索一些
- 后面信息越来越多，逐渐减少探索

你的代码里写成了：

```python
epsilon = 1. / self.total_count
```

也就是随着总步数增加，`epsilon` 反比例衰减。

这样做的直觉是：

- 前期多试试
- 后期更相信已有统计结果

所以它通常比固定 `epsilon` 更符合“先探索、后利用”的思路。

不过它也不是一定完美，因为：

- 衰减太快，可能过早停止探索
- 衰减太慢，又会浪费很多步数

## 13. UCB 在做什么

UCB 是 `Upper Confidence Bound`，中文通常叫上置信界算法。

它的关键思想不是简单随机探索，而是：

`优先探索那些“当前均值看起来不错，而且不确定性还比较大”的动作`

你的代码里核心公式是：

```python
ucb = self.estimates + self.coef * np.sqrt(
    np.log(self.total_count) / (2 * (self.counts + 1))
)
```

这个式子可以分成两部分：

- `self.estimates`：当前这根拉杆的平均表现
- 后面那一项：探索奖励，也可以理解成“不确定性补偿”

所以 UCB 选的不是“均值最高的臂”，而是：

`均值 + 探索加成` 最高的臂

## 14. 为什么 UCB 的探索项长这样

UCB 的探索项是：

```python
sqrt(log(t) / N(a))
```

直觉上：

- 总步数 `t` 越大，说明问题被研究得越久，对置信要求越高
- 某根臂被尝试次数 `N(a)` 越少，不确定性越大

所以：

- 被试得很少的臂，会得到更大的探索奖励
- 被试得很多的臂，探索奖励会逐渐变小

这比 `epsilon-greedy` 更“有针对性”，因为它不是盲目随机，而是在定向补足不确定性。

## 15. 为什么 UCB 代码里要写 `self.counts + 1`

这是一个很容易卡住，但其实很重要的实现细节。

如果不写 `+ 1`，当某根臂还没被尝试过时：

```python
self.counts[k] = 0
```

分母就会变成 0，导致除零问题。

所以这里加 `1` 的作用主要是：

- 防止除零
- 让没有被试过的臂也能算出 UCB 值
- 让前期探索逻辑更容易实现

不过要注意：

- 这更像是工程实现上的平滑处理
- 标准理论推导里通常会先保证每根臂至少被选过一次

## 16. Thompson Sampling 在做什么

Thompson Sampling 是我觉得特别有意思的一种方法，因为它不是直接对均值做贪心，也不是手工加一个置信上界，而是：

`把每根拉杆当前“可能有多好”看成一个概率分布，再从这个分布里采样做决策`

在你的代码里：

```python
self.a = np.ones(bandit.k)
self.b = np.ones(bandit.k)
samples = np.random.beta(self.a, self.b)
k = np.argmax(samples)
```

这里的意思是：

- 每根拉杆维护一个 Beta 分布
- `a` 可以理解成成功次数的累计
- `b` 可以理解成失败次数的累计
- 每一步都从每根臂对应的分布里随机采样一个“可能的中奖率”
- 选择采样值最大的臂

## 17. 为什么 Thompson Sampling 很自然

Thompson Sampling 的探索不是硬塞进去的，而是自动长出来的。

因为：

- 如果某根臂数据很少，它的后验分布会更宽
- 分布更宽，就意味着它偶尔可能采出一个很大的值
- 于是它自然会被探索到

而如果某根臂已经被反复尝试，分布会越来越集中：

- 好臂会稳定采出大值
- 差臂会稳定采出小值

所以 Thompson Sampling 的直觉可以概括成：

`按“它成为最优臂的可能性”来分配探索机会`

## 18. Beta 分布为什么适合 Bernoulli Bandit

这是一个很经典的配对：

- 奖励服从 Bernoulli 分布
- 先验和后验用 Beta 分布

这样做的好处是：

- 数学形式很方便
- 每次观察到一次成功或失败后，参数更新特别简单

你的代码里更新方式是：

```python
self.a[k] += r
self.b[k] += 1 - r
```

也就是说：

- 如果这次奖励是 `1`，成功次数加一
- 如果这次奖励是 `0`，失败次数加一

这个更新非常直观，也很适合入门理解贝叶斯方法。

## 19. 结合自己的代码梳理一遍

在 `mab.py` 里，整体流程大致是：

1. 随机生成一个 `K` 臂 Bernoulli Bandit
2. 记录每根臂的真实中奖概率
3. 让不同策略各自运行若干步
4. 每一步更新奖励估计和累计懊悔值
5. 最后画出累计懊悔曲线

对应的公共框架在 `Solver` 里：

```python
def run(self, num_steps):
    for _ in range(num_steps):
        k = self.run_one_step()
        self.counts[k] += 1
        self.actions.append(k)
        self.update_regret(k)
```

所以这段代码实际上做了两件分离得很清楚的事：

- `run_one_step()` 负责“当前这一步该选谁”
- `run()` 负责“记录动作、次数和懊悔值”

这种写法很适合后面继续加新策略。

## 20. 为什么单次实验结果差距会很大

这是我在学习这部分时一个特别需要注意的点。

多臂老虎机本身带随机性，主要体现在两层：

- 环境的奖励是随机的
- 策略自身的探索过程也可能是随机的

所以即使是同一个算法：

- 不同随机种子下，结果也可能差很多
- 有时会很快找到最优臂
- 有时会早期误判，然后卡很久

这也是为什么只看一次输出，容易误以为“算法好像有 bug”。

更合理的比较方式通常是：

- 多跑几次
- 看平均累计懊悔
- 或者看多次实验的均值曲线

## 21. 多臂老虎机和后面强化学习的关系

多臂老虎机虽然没有状态转移，但它已经把强化学习里很重要的一部分问题抽出来了：

- 奖励不确定
- 决策要边试边学
- 不能只凭眼前结果下结论
- 探索和利用必须平衡

后面到了 Q-learning、DQN，其实只是把问题进一步升级成：

- 不只是“选哪根臂”
- 而是“在不同状态下该选什么动作”

所以可以把它们的关系理解成：

- 多臂老虎机：没有状态转移的决策问题
- Q-learning / DQN：有状态转移的序列决策问题

## 22. 这阶段我最该记住的几句话

- 多臂老虎机的核心不是环境复杂，而是探索和利用的矛盾很纯粹
- 每根拉杆的单次奖励是随机的，算法真正要学的是长期期望收益
- 累计懊悔值衡量的是“因为没有一直选最优动作而损失了多少期望收益”
- `epsilon-greedy` 简单直接，但探索比较粗糙
- 衰减 `epsilon` 的想法是前期多探索、后期多利用
- UCB 会优先探索“不确定但可能好”的动作
- Thompson Sampling 通过后验分布采样，把探索和利用自然结合了起来

## 23. 我目前最容易记错的点

### 错误 1：懊悔值就是没中奖的次数

不对。

正确说法是：

懊悔值比较的是“当前所选动作的期望奖励”和“最优动作的期望奖励”的差距，不是单次实际奖励差距。

### 错误 2：`epsilon-greedy` 里的探索就是乱选，所以没什么道理

不对。

正确说法是：

它虽然简单，但本质上是在给未知动作保留尝试机会，避免过早陷入局部最优。

### 错误 3：UCB 后面那个探索项只是为了把数值变大一点

不对。

正确说法是：

探索项表达的是不确定性补偿，被尝试越少的臂，不确定性越高，因此要额外鼓励探索。

### 错误 4：Thompson Sampling 完全靠随机，所以不如 UCB 稳

不对。

正确说法是：

Thompson Sampling 的随机性不是盲目随机，而是基于后验分布的不确定性来随机，很多场景下效果非常强。
